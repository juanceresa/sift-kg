## 1. 项目背景与sift-kg价值

如果你打算从头构建一个知识图谱系统，sift-kg 提供了一个非常优秀的参考实现。它解决了从非结构化文档集合构建知识图谱的核心问题：

- **多格式文档输入** → 纯文本提取
- **schema定义** → 支持自动发现和预定义两种模式
- **LLM抽取** → 实体关系并行抽取，成本控制
- **去重消歧** → 三层人机协作架构平衡速度和准确性
- **可视化与导出** → 开箱即用的探索体验

sift-kg 的核心优势是**零配置、模块化、人机协作**，适合个人研究者、调查工作、文献分析等场景。你可以基于这套架构思想，根据自己的需求做适应性改造。

## 2. 核心架构思想借鉴

### 2.1 流水线架构（Pipeline Architecture）

sift-kg 采用经典的流水线处理模式，每个模块只做一件事，职责清晰：

```
文档集合 → 文本提取 → 分块 → Schema发现 → 抽取 → 图谱构建 → 实体消歧 → 叙事生成 → 可视化/导出
```

**为什么这么设计？**

- 每个阶段可以独立测试和优化
- 便于增量处理（只重新处理修改过的文档）
- 可以单独使用某个模块（比如只用抽取，不用自己的存储）
- 易于调试，每个阶段的输出都可以检查

**你的实现参考**:
- 保持阶段之间解耦，通过文件/标准数据格式交换
- 每个阶段保存中间结果，支持断点续跑
- 缓存昂贵的LLM调用结果，避免重复花钱

### 2.2 人机协作而非纯自动化

sift-kg 的核心设计原则：**LLM提出建议，人做最终决定**。从不自动合并实体。

这个设计非常重要：
- LLM一定会犯错，特别是实体消歧
- 自动合并错误会污染整个图谱，很难修复
- 对于关键应用（法律、调查、族谱）准确性比速度重要
- 用户可以随时调整决策，过程可追溯

**你的实现参考**:
- 关键决策（实体合并、低置信度关系）留给人工审核
- 保存审核决策历史，支持增量处理
- 提供多种审核方式（交互式UI/YAML编辑）满足不同精度需求

### 2.3 零配置理念

sift-kg 设计目标是**一条命令就能开始工作**:
- 不需要用户提前搭建数据库
- 默认schema-free模式，LLM自动发现适合语料的schema
- 合理的默认配置，用户只需改少数参数
- 本地处理，文档不出机器，隐私友好

**你的实现参考**:
- 提供合理默认值，让新手能快速跑通
- 支持渐进式配置，从默认到精细调整
- 不要强制用户事前定义完整schema

## 3. 模块化流水线设计

### 3.1 模块职责划分参考

sift-kg 的模块划分非常清晰，你可以直接借鉴：

| 模块 | 职责 | 关键设计点 |
|------|------|-----------|
| `config` | 配置管理 | 优先级：CLI > 环境变量 > .env > 配置文件 > 默认值 |
| `ingest` | 文档摄取 | 多格式提取，OCR支持，分块 |
| `domains` | 域配置 | schema定义，预打包域，自动发现 |
| `extract` | 实体关系抽取 | 异步并发，增量缓存，一次调用抽取实体+关系 |
| `graph` | 知识图谱核心 | 数据结构，预去重，社区检测 |
| `resolve` | 实体消歧 | LLM建议，人工审核，图手术应用合并 |
| `narrate` | 叙事生成 | 社区标签，实体描述，报告生成 |
| `visualize` | 可视化 | 交互式浏览器查看 |
| `export` | 导出 | 多种标准格式输出 |

## 4. 关键设计决策参考

### 4.1 为什么用 NetworkX + JSON 而不是数据库？

sift-kg 选择：
- NetworkX 图数据结构
- JSON 文件存储

原因：
- 项目定位是**个人文档集合**（千级别实体），不需要数据库
- 零配置：用户不需要安装运行数据库
- 单文件存储便于版本控制和分享
- NetworkX 成熟稳定，内置算法丰富，导出支持完善

**当你的需求不同时，可以考虑**:
- 需要支持十万+实体 → 考虑 Neo4j / PostgreSQL + 扩展
- 需要多用户并发访问 → 需要数据库
- 需要复杂查询 → 数据库更好

但对于个人使用和中小型项目，NetworkX+JSON 非常棒。

### 4.2 实体ID设计

sift-kg 的实体ID格式：`{entity_type}:{normalized_name}`

示例：`person:samuel_benjamin_bankman_fried`, `concept:transformer`

优势：
- **确定性**: 相同名称 → 相同ID，自动合并
- **可读性**: ID本身包含类型和名称信息
- **可预测**: 易于手动构造查询

这是一个非常简洁有效的设计，推荐直接借鉴。

### 4.3 关系规范化

关系的规范键：`{source_id}|{relation_type}|{target_id}`

优势：
- 同一三元组多次提及自动聚合到同一条边
- 置信度自动聚合（独立证据互相增强）
- 累积支持计数和来源文档

这解决了"同一关系被多次提及"的问题，避免图中产生大量冗余边。

### 4.4 三层实体去重

sift-kg 采用三层去重，平衡成本和准确性：

| 层级 | 方法 | 成本 | 作用 |
|------|------|------|------|
| 第一层 | 确定性预去重 | 免费 | 处理明显重复（Unicode变体、单复数、头衔变体） |
| 第二层 | LLM批量比较 | 中等成本 | 提出可能需要合并的候选 |
| 第三层 | 人工审核批准 | 用户时间 | 做最终决定 |

预去重具体步骤：
1. Unicode 规范化 (NFC)
2. 去除常见头衔前缀（35+种）
3. 单复数归一化
4. SemHash 模糊匹配（阈值0.95）捕捉近似拼写差异

这个设计受到 KGGen (NeurIPS 2025) 论文启发，实践效果很好。**强烈推荐借鉴**。

### 4.5 置信度聚合策略

sift-kg 默认使用 `product_complement` 策略：

```
结果 = 1 - Π(1 - cᵢ) 对所有提及 i
```

理论基础：独立弱证据互相增强。比如三个独立文档都提及同一关系，每次置信度0.7，结果是：

```
1 - (0.3 * 0.3 * 0.3) = 0.973
```

也支持 `mean`（平均）和 `max`（最大）供选择。这个聚合策略非常合理，推荐借鉴。
