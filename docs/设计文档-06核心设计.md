## 4. 核心数据结构

### 实体ID格式

实体ID遵循模式: `{entity_type}:{normalized_name}`

- 全小写
- 空格替换为下划线
- 示例: `person:samuel_benjamin_bankman_fried`, `organization:openai`, `concept:transformer_architecture`

这种设计保证:
- **确定性**: 相同名称→相同ID，自动合并
- **可读性**: 从ID就能知道类型和名称
- **可预测**: 易于手动构造查询

### 关系规范键

关系的规范键: `{source_id}|{relation_type}|{target_id}`

保证: 同一三元组(source, relation_type, target)多次提及自动聚合到同一条边，累积置信度和支持证据。

### JSON存储格式

```json
{
  "metadata": {
    "created_at": "2026-03-30T12:34:56",
    "updated_at": "2026-03-30T12:34:56",
    "entity_count": 425,
    "relation_count": 890,
    "document_count": 12,
    "entity_type_summary": {
      "PERSON": 150,
      "ORGANIZATION": 80
    },
    "sift_kg_version": "0.9.0"
  },
  "nodes": [
    {
      "id": "person:john_doe",
      "entity_type": "PERSON",
      "name": "John Doe",
      "confidence": 0.95,

      "attributes": {}
    }
  ],
  "links": [
    {
      "source": "person:john_doe",

      "support_documents": ["document1", "document2"],

          "source_document": "document1",
          "confidence": 0.92,

```

### 合并建议格式 (YAML)

```yaml
proposals:
- canonical_id: person:samuel_benjamin_bankman_fried
  canonical_name: Samuel Benjamin Bankman-Fried
  entity_type: PERSON
  status: DRAFT
  members:
  - id: person:bankman_fried
    name: Bankman-Fried
    confidence: 0.99
  reason: Same person referenced with full name vs. surname only.
```

状态: `DRAFT` (待审核), `CONFIRMED` (批准合并), `REJECTED` (拒绝合并)

## 5. 核心算法

### 5.1 置信度聚合

sift-kg支持三种置信度聚合策略，用于同一关系多次提及:

1. **product_complement** (默认):
   ```
   结果 = 1 - Π(1 - cᵢ) for all mentions i
   ```
   理论基础: 独立弱证据互相增强。每个额外提及都增加我们对关系的信心。如果一个关系被三个独立文档提及，每次置信度0.7，结果是:
   ```
   1 - (0.3 * 0.3 * 0.3) = 1 - 0.027 = 0.973
   ```

2. **mean**: 简单算术平均，适合相关证据。

3. **max**: 取最大值，适合只关心最高置信度。

### 5.2 三层实体去重

sift-kg采用三层去重架构，平衡成本和准确性:

| 层级 | 方法 | 成本 | 作用 |
|------|------|------|------|
| 第一层 | 确定性预去重 | 免费 | 处理明显重复 (Unicode变体、单复数、头衔变体) |
| 第二层 | LLM批量比较 | 中等成本 | 提出可能需要合并的候选 |
| 第三层 | 人工审核批准 | 免费（用户时间）| 做最终决定 |

**预去重具体步骤**:
1. Unicode规范化 (NFC)
2. 去除常见头衔前缀 (Dr., Mr., Mrs., Senator, Judge, 等35+种)
3. 单复数归一化 (inflect库)
4. SemHash模糊字符串匹配，阈值0.95，捕捉近似拼写差异

这种设计受到KGGen (NeurIPS 2025)论文启发。

### 5.3 社区发现

使用Louvain算法进行社区检测，识别图中密集连接的子图:

- 每个实体分配到恰好一个社区
- 用于可视化中给不同社区着不同颜色
- 用于叙事生成中按社区分组实体描述
- 帮助AI理解知识的拓扑结构（哪些主题聚在一起）

## 6. 完整执行流程

### 典型用户工作流

```
1. sift init                      → 创建sift.yaml和.env.example
2. 编辑.env添加API密钥
3. sift extract ./documents/     → 抽取所有文档，缓存结果
4. sift build                     → 构建图谱，预去重，社区检测
5. sift resolve                  → LLM提出合并建议
6. sift review                   → 人工审核（可自动批准高置信度）
7. sift apply-merges             → 应用合并，修改图谱
8. sift narrate                  → 生成叙事报告
9. sift view                     → 在浏览器中交互式浏览
10. sift export graphml          → 导出到外部工具分析
```

### 增量处理流程

sift-kg支持增量更新知识库:

1. 添加新文档到文档目录
2. `sift extract` 只重新提取修改过的文档，复用已有提取结果
3. `sift build` 重新构建整个图谱
4. `sift resolve` 只寻找新的重复，保留之前审核决策
5. `sift apply-merges` 重新应用所有已确认合并

这大幅降低了迭代成本，允许知识库随时间增长。

### 抽取阶段执行流程

```
输入: 文档目录
输出: output/extractions/*.json, output/discovered_domain.yaml

对于每个文档:
  如果已抽取且配置未变 → 跳过
  否则 → 读取文档文本（带可选OCR）→ 分块 → 抽取

schema-free模式:
  如果没有已发现schema → LLM发现schema → 保存
  使用发现的schema抽取

对每个块:
  生成prompt（注入文档上下文 + 域定义）→ LLM调用 → 解析JSON → 去重 → 返回
```

### 实体合并应用流程

```
输入: 图谱，merge_proposals.yaml
输出: 修改后的图谱

对于每个确认的合并:
  对于每个成员实体:
    合并成员属性到规范实体（来源文档合并，置信度取高）
  对于每条边:
    如果边指向/来自成员 → 重写指向规范实体
    如果产生自环 → 移除
  移除成员节点
```

## 7. 设计决策

### 为什么使用NetworkX?

- 成熟稳定，Python生态最好的图处理库
- 对于中小型图谱（千级别实体）完全够用
- 内置多种图算法（社区检测等）
- 导出多种格式支持完善
- 不需要数据库，单文件JSON足够简单

项目定位是"零配置"，不需要用户搭建数据库 infrastructure。使用NetworkX+JSON符合这个定位。

对于**十万级以上实体**，NetworkX可能内存不足，但这超出了sift-kg的目标用例（个人文档集合，研究项目，调查）。

### 为什么人机协作?

LLM会犯错。在知识图谱构建，特别是实体合并这种关键步骤，错误会污染整个图谱。sift-kg设计原则是: **LLM提出建议，人做决定**。从不自动合并。

对于高精度用例（法律、调查、族谱）这是必须的。对于快速探索，用户可以设置自动批准高置信度合并平衡速度和准确性。

### 为什么schema-free默认?

很多时候你事前不知道文档集合里有什么类型的实体和关系。让LLM先看一眼，设计适合的schema比强迫用户事前定义好所有类型更方便。用户可以事后编辑调整发现的schema。

当然，如果用户有明确的分类体系，也支持预定义schema封闭模式。

### 为什么异步并发?

LLM调用是延迟瓶颈，异步并发可以让多个LLM请求同时处理，大幅减少总等待时间。所有块跨文档共享并发限制，避免多个文档处理时并发溢出。

### 为什么单次调用抽取实体和关系?

分开两次调用（先实体后关系）会让LLM更准确，但成本加倍。实践中，现代LLM足够强大，可以在一次调用中同时抽取实体和关系，节省时间和成本。如果抽取失败，该块返回空，不会崩溃整个管道。

## 9. 扩展性设计

### 添加新的文本提取后端

1. 在 `ingest/base.py` 实现 `TextExtractor` 协议
2. 在 `reader.py` `read_document()` 添加选择逻辑
3. 更新配置验证 `config.py`

遵循协议模式，不需要修改其他代码。

### 添加新的OCR后端

1. 在 `ocr.py` 添加后端实现
2. 更新验证 `config.py` 中的有效后端列表
3. 在 `reader.py` 传递参数

### 添加新的导出格式

1. 在 `export.py` 添加新的导出函数
2. 在 `cli.py` `export` 命令添加格式选项
3. 使用NetworkX已有导出功能通常只需要几十行代码

### 自定义域

不需要修改代码，用户可以创建自己的 `domain.yaml` 文件:

```yaml
name: my-custom-domain
fallback_relation: RELATED_TO
entity_types:
  MY_TYPE:
    description: Description of my type
    extraction_hints:
      - Look for these entities...
relation_types:
  MY_RELATION:
    description: Description of relation
    source_types: [MY_TYPE]
    target_types: [OTHER_TYPE]
```

然后使用: `sift extract ./docs --domain path/to/domain.yaml`

## 项目总结

sift-kg 是一个设计精良的模块化知识图谱构建工具，核心优势在于:

1. **零配置上手**: 不需要数据库，不需要基础设施，一条命令就能开始
2. **灵活性**: 支持预定义schema和自动发现两种模式，满足不同用例
3. **准确性**: 三层去重+人机协作，保证最终图谱质量
4. **可扩展性**: 模块化设计，易于添加新的后端和格式
5. **AI集成**: 天生设计为AI agent的第二大脑，提供结构化持久记忆

项目适合个人研究者、调查人员、数据记者、任何人想要从文档集合中提取结构化知识。
