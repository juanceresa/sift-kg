### 3.3 抽取模块 `extract/`

**职责**: 从分块后的文本中抽取实体和关系，处理整个文档集合。

**组件**:

1. **`models.py` - 数据模型**
   - `ExtractedEntity`: 抽取的实体，包含名称、类型、属性、置信度、上下文
   - `ExtractedRelation`: 抽取的关系，包含类型、源实体、目标实体、置信度、证据文本
   - `DocumentExtraction`: 单个文档的抽取结果，包含所有实体和关系，以及元数据（模型、域、分块大小、时间戳）

2. **`extractor.py` - 抽取编排**
   - **异步并发**: 使用 `asyncio.Semaphore` 控制并发LLM调用数（默认4）平衡速度和速率限制
   - **增量处理**: 检查现有提取结果是否过期（模型/域/分块大小变化才重新提取）
   - **文档上下文**: 对每个文档，用第一个块生成2-3句文档摘要，注入到后续所有块的prompt中，帮助LLM理解全局上下文
   - **schema发现**: 在schema-free模式下，先用LLM从文档样本发现适合这个语料的实体和关系类型，保存为YAML供后续重用
   - **文档内去重**: 同一文档内同名同类型实体只保留最高置信度的一个，但收集所有唯一上下文

   **主要函数**:
   - `extract_all()`: 处理多个文档，共享并发限制
   - `extract_document()`: 处理单个文档，保存结果到JSON
   - `extract_from_text()`: 从纯文本抽取（无文件I/O，可测试）
   - `_aextract_chunk()`: 异步抽取单个块，一次LLM调用同时抽取实体和关系

3. **`llm_client.py` - LLM客户端**
   - 封装 LiteLLM 提供统一接口
   - 支持所有LiteLLM兼容的模型提供商
   - 跟踪总美元成本用于预算控制
   - 提供同步和异步调用
   - JSON模式解析，处理解析失败回退

4. **`prompts.py` - 提示词构造**
   - 根据域配置动态构造prompt
   - 注入实体和关系类型定义、提取提示
   - 注入文档上下文
   - 强制JSON输出格式

**关键设计决策**:
- **单次调用抽取**: 实体和关系在一次LLM调用中抽取，而不是两次分开调用，节省成本和延迟
- **并发设计**: 所有块跨文档共享同一个信号量，避免多个文档之间浪费并发槽
- **增量缓存**: 只重新提取真正需要更新的文档，大幅降低重复运行的成本

### 3.4 域配置系统 `domains/`

**职责**: 定义抽取schema，控制抽取哪些类型的实体和关系。

**核心模型**:

```python
class EntityTypeConfig(BaseModel):
    description: str                    # 类型描述，注入prompt
    extraction_hints: list[str]        # 提取提示，帮助LLM理解
    canonical_names: list[str]          # 封闭词汇表，只有这些值允许
    canonical_fallback_type: str | None # 不匹配时回退到的类型

class RelationTypeConfig(BaseModel):
    description: str
    source_types: list[str]            # 允许的源实体类型
    target_types: list[str]            # 允许的目标实体类型
    symmetric: bool = False            # 是否对称关系
    extraction_hints: list[str]
    review_required: bool = False      # 是否需要人工审核

class DomainConfig(BaseModel):
    name: str
    version: str = "1.0.0"
    description: str
    entity_types: dict[str, EntityTypeConfig]
    relation_types: dict[str, RelationTypeConfig]
    system_context: str | None        # 全局系统上下文注入prompt
    fallback_relation: str | None      # 不匹配时的回退关系类型
    schema_free: bool = False          # 是否schema-free模式
```

**预打包域**:

| 域名 | 用途 | 主要实体类型 |
|------|------|-------------|
| `schema-free` | 默认模式 | LLM自动从语料发现 |
| `general` | 通用文档分析 | PERSON, ORGANIZATION, LOCATION, EVENT, DOCUMENT |
| `osint` | 调查和开源情报 | SHELL_COMPANY, FINANCIAL_ACCOUNT, OFFSHORE_JURISDICTION等 |
| `academic` | 文献综述 | CONCEPT, THEORY, METHOD, SYSTEM, FINDING, RESEARCHER等 |

**schema-free模式工作原理**:
1. 第一次运行时，LLM从5个文档样本中分析内容
2. LLM设计适合该语料的实体类型和关系类型
3. 保存到 `output/discovered_domain.yaml`
4. 后续运行重用发现的schema，保持一致性
5. 用户可以手动编辑调整

**schema强制特性**:
- 如果定义了封闭schema，LLM被指令只能使用定义好的类型
- 如果设置了 `fallback_relation`，不匹配的关系会映射到回退类型
- 实体类型的 `canonical_names` 强制封闭词汇表，不匹配会自动重新分类
- `canonical_names` 适合分类学、预定义分类体系

**加载器 `loader.py`**:
- 支持从文件路径加载自定义域
- 支持按名称加载预打包域
- 处理yaml配置到pydantic模型的转换
